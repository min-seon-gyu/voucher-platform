# 스케일 스터디 — 실측 결과 (1차)

> **측정일 기준 1차 실행.** 단일 인스턴스 합성 부하테스트. 절대 수치는 이 하드웨어/설정 한정이며 "프로덕션 트래픽"으로 일반화하지 않는다. 신뢰 가능한 주장은 둘: ① **동일 하드웨어에서 병목 제거로 상대 처리량 개선**, ② **포화 전 구간에서 거래 정합성 유지**.

## 측정 환경
- 단일 Windows 노트북, Docker Desktop. **app + MySQL + Redis + 관측스택 + k6가 모두 같은 머신에 공존** → 테스트 리그 자체가 병목에 기여(아래 한계 참조).
- 앱: Spring Boot 3.2 (JDK 17), HikariCP 풀 **10 → 40** (튜닝 변수). MySQL 8, Redis 7.
- 부하: k6 open model(ramping-arrival-rate), 응답을 success/lock_timeout/insufficient/limit/5xx로 분류.

## 시나리오별 결과

### A. 단일 상품권 핫키 — 락 직렬화 상한
- 한 상품권에 모든 결제를 집중(인위적 최악). offered 1,200 rps.
- 성공 ~14/s + **lock_timeout ~10/s**(Redisson 5s 대기 후 503, 의도된 백프레셔), **server_error(5xx) = 0**.
- p99가 60s(클라이언트 타임아웃)까지 상승, `dropped_iterations` 312/s → 모든 요청이 단일 락에 직렬화되어 처리량 천장 형성.
- **결론**: 핫키는 락으로 정합성을 지키는 대가로 직렬화 천장이 존재하며, 포화 시 **5xx 없이 락 타임아웃/큐잉으로 graceful 하게 백프레셔**된다(크래시 0).

### B. 분산 다수 상품권 — HikariCP 풀 튜닝 before/after  ⭐핵심
동일 부하(offered 600 rps, 분산 150 voucher)에서 **풀 크기만 단일 변수로 변경**:

| 지표 | 풀 10 (before) | 풀 40 (after) | 변화 |
|---|---|---|---|
| **성공 처리량** | 11.6 /s | **57.4 /s** | **≈ 5배** |
| 성공 요청 p95 | 96 ms | 186 ms | ↑ |
| 실패율(타임아웃 포함) | 61.6% | 21.4% | ↓ |
| server_error(5xx) | 16 | 117 | ↑ |

- **병목 이동(두더지잡기)**: 풀 10에서는 **HikariCP 풀 고갈**이 처리량을 막았고, 풀 40으로 늘리자 처리량이 5배로 뛰었으나 **이번엔 DB가 새 병목**이 됨(성공 지연 p95 96→186ms, DB 압박으로 5xx 비율 0.26%→0.85%).
- **결론**: 풀은 처리량 노브가 아니라 **DB가 감당할 동시성의 게이트**. 한 병목을 제거하면 다음 병목(DB)으로 이동한다.

### C. 동일 지역 동시 발행 — Redis Lua 카운터 + 월한도 정확성
- 50개 멤버가 동일 지역(월한도 1e8, 건당 1e4)에 동시 발행. 발행 성공 p95 **138 ms**.
- 한도 도달 후 초과분은 `REGION_MONTHLY_LIMIT_EXCEEDED`로 차단.
- **정확성 검증(SQL)**: 해당 지역 발행 `10,000건 = 정확히 100,000,000원 = 한도 그대로`, **과다발행 0건**. → Redis Lua 원자 연산(INCRBY+검증+롤백)이 **동시성하에서도 한도를 정확히 보장**.

## 정합성 검증 (전체 부하 종료 후, SQL)
누적 **결제 21,258건 + 발행 10,303건** 이후:

| 불변식 | 결과 |
|---|---|
| 복식부기 전역 균형 (DEBIT 합 == CREDIT 합) | **2,031,102,125,800.00 == 2,031,102,125,800.00 ✅** |
| 음수 잔액 voucher | **0건 ✅** |
| 지역 월한도 과다발행 | **0건 ✅** |

→ **고부하·포화 구간을 거친 뒤에도 거래 정합성(복식부기 불변식·잔액 음수 불가·한도 정확성)이 깨지지 않음.**

## 정직한 한계
- 실 사용자 0의 **단일 인스턴스 합성 부하**. app·DB·Redis·k6가 한 노트북에 공존해 **테스트 리그 자체가 포화**(`dropped_iterations` 다수, p99 60s 타임아웃)했고, 따라서 **절대 TPS는 이 환경의 천장**이지 앱의 한계가 아님.
- 신뢰 가능한 주장은 둘로 한정: **① 동일 하드웨어에서 풀 튜닝으로 성공 처리량 ≈5배 상대 개선, ② 포화 전 구간 정합성 유지.**
- 더 정확한 절대 수치를 원하면 k6를 별도 머신으로 분리하고 DB를 네이티브 디스크로 옮겨 재측정 필요.

## 포트폴리오용 (실측 반영)

**한 줄**
> Kotlin/Spring 상품권 금융 백엔드에 Prometheus/Grafana 관측 스택을 구축하고 k6 오픈모델 합성 부하로 병목을 진단 — **HikariCP 풀 10→40 튜닝으로 동일 하드웨어 성공 처리량 ≈5배(11.6→57.4 redeem/s) 개선**하고 병목이 풀→DB로 이동함을 확인, **포화 구간에서도 복식부기 불변식(DEBIT==CREDIT)·잔액 음수 0·월한도 과다발행 0을 SQL 어서션으로 유지 입증** *(단일 인스턴스 synthetic load study)*.

**핵심 메시지 3개**
1. 병목을 측정→진단→처방→재측정 루프로 이동시켰다(풀→DB). 절대치가 아닌 **상대 개선**으로 정직하게 보고.
2. 고부하에서도 **정합성 불변식이 깨지지 않음**을 데이터로 증명(DEBIT==CREDIT, 음수 0, 과다발행 0).
3. 포화는 **5xx 크래시가 아니라 락 타임아웃/큐잉으로 graceful** 하게 처리됨(백프레셔 설계 검증).
